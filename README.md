# Wazuh SIEM Home Lab — Attack Detection & Monitoring

## Objective
Deploy Wazuh SIEM in a home lab environment to simulate 
real-world SOC operations — including attack simulation, 
custom detection rule creation, alert triage, and MITRE 
ATT&CK mapping.

---

## Lab Environment

| Component | Details |
|-----------|---------|
| Wazuh Manager | Ubuntu 22.04 (VM) |
| Monitored Agent | Windows 10 / Kali Linux (VM) |
| Hypervisor | VirtualBox |
| Network | Host-only / NAT network between VMs |
| Wazuh Version | 4.x |

---

## Deployment Steps

### Step 1 — Install Wazuh Manager
```bash
curl -sO https://packages.wazuh.com/4.7/wazuh-install.sh
sudo bash wazuh-install.sh -a
```
After installation, Wazuh dashboard is accessible at:
`https://YOUR-MANAGER-IP` — default credentials set during install.

### Step 2 — Deploy Wazuh Agent (on monitored machine)
On the agent machine, download and install the Wazuh agent 
package for your OS. Register it to point to the manager IP:

```bash
# Linux agent registration
sudo /var/ossec/bin/agent-auth -m MANAGER_IP
sudo systemctl start wazuh-agent
```

For Windows: download the MSI installer from the Wazuh 
dashboard, run it, enter manager IP when prompted.

### Step 3 — Verify Agent Connection
On the manager, confirm agent is active:
```bash
sudo /var/ossec/bin/agent_control -l
```
Active agents will show status: `Active`

### Step 4 — Configure Log Collection
Edit `/var/ossec/etc/ossec.conf` on the manager to collect 
logs from relevant sources:

```xml

  
    syslog
    /var/log/auth.log
  
  
    syslog
    /var/log/syslog
  

```

## Custom Detection Rules

All custom rules are stored in `/detection-rules/` folder 
in this repository. They are deployed on the Wazuh manager 
at `/var/ossec/etc/rules/local_rules.xml`.

### Rule 1 — SSH Brute Force Detection
Triggers when more than 5 failed SSH login attempts occur 
within 60 seconds from the same source IP.

```xml

  5716
  
  SSH Brute Force Attack Detected
  
    T1110
  
  brute_force,authentication_failed

```

### Rule 2 — Reverse Shell Detection
Triggers when a shell process (bash/sh) is spawned by an 
unexpected parent process — key indicator of exploitation.

```xml

  syscheck
  bash|sh|cmd
  ppid=(?!bash|sh|terminal|sshd)\S+
  Possible Reverse Shell — Shell spawned 
  by non-shell parent process
  
    T1059
  
  reverse_shell,execution

```

### Rule 3 — Privilege Escalation Detection
Triggers on sudo usage by non-administrative accounts or 
failed sudo attempts indicating escalation attempts.

```xml

  5402
  Possible Privilege Escalation — 
  Repeated sudo failures detected
  
    T1068
  
  privilege_escalation,sudo

```

## SOC Analyst Workflow — How I Triage These Alerts

This section documents how I approach alert triage — the 
core daily function of a Tier 1 SOC Analyst.

### Step 1 — Receive Alert
Alert arrives in Wazuh dashboard with severity level, 
rule ID, and source/destination details.

### Step 2 — Initial Classification
- What is the rule that fired?
- What asset is affected?
- What is the severity level (1-15 in Wazuh)?
- Is this a known asset or unknown source IP?

### Step 3 — Investigate
For brute force alert:
- Check source IP against threat intel (VirusTotal, AbuseIPDB)
- Verify if any login succeeded after the failures
- Check if source IP has hit other assets

For reverse shell alert:
- Identify parent process and child process
- Check if the spawning process is legitimate
- Isolate the endpoint if confirmed malicious

### Step 4 — Escalate or Close
- **False Positive** → Document reason, tune rule if needed
- **True Positive** → Escalate to Tier 2 with full findings
- **Suspicious** → Continue monitoring, add to watchlist

### Step 5 — Document
Every alert gets a ticket entry with: timestamp, rule fired,
affected asset, investigation steps taken, and final 
disposition.

---

## Key Learnings

- Wazuh rule logic uses frequency and timeframe thresholds 
  to reduce false positives on noisy log sources
- Process lineage monitoring (parent → child process 
  relationships) is one of the highest-fidelity detection 
  methods available in endpoint monitoring
- MITRE ATT&CK mapping on every rule makes escalation 
  conversations with Tier 2 and IR teams significantly 
  faster
- Alert fatigue is real — rule tuning is as important as 
  rule creation in a production SOC environment
